* Introduction

Eve is an Emacs major mode for editing spoken-word videos by manipulating text
instead of cutting timelines manually.

** The Big Idea

Traditional video editing works at the frame level: you scrub a timeline, find
a cut point, and trim.  When your source material is a talking-head recording
or a podcast, almost every edit decision is driven by what was /said/ — not by
what is visible.

Eve flips the model.  A Textual Join Manifest (=.tjm.json=) stores your video
as a sequence of transcribed segments, each with per-word timestamps.  You
read and edit the transcript in Emacs.  Delete a word, and it disappears from
the rendered video.  Reorder paragraphs, and the clips follow.  Annotate a
segment with a b-roll clip, and the renderer splices it in.  The text is the
edit.

** The Core Workflow

The typical eve session looks like this:

1. *Transcribe* — run =M-x eve-transcribe= on a directory of raw recordings.
   Eve calls =eve transcribe= and opens the resulting =.tjm.json= in eve-mode.

2. *Edit* — delete filler words (=d=, =f=, =F=), reorder segments (=J=, =K=),
   add markers to name sections (=o=), attach b-roll (=b=).

3. *Preview* — press =C-c SPC= to play the source clip from point, or
   =C-c C-c= to compile a draft video in seconds.

4. *Compile* — =C-c C-c= renders the manifest as an =.mp4=.  Draft renders
   are fast (mjpeg, parallel, cached).  =C-u C-c C-c= produces a final
   delivery-quality H.264 export.

5. *Iterate* — the segment cache means only changed segments re-encode.
   Speculative pre-rendering warms the cache on every save, so pressing
   =C-c C-c= after a small edit is nearly instant.

** About This Manual

This manual covers the Emacs package (=eve.el=) and the accompanying CLI
(=eve text-edit=, =eve transcribe=).  The normative TJM v1.1 format
specification is in =docs/tjm-spec.md=; this manual cross-references it but
does not reproduce it in full.

** Installation

*** Requirements

- *Emacs* 29.1 or later
- *mpv* for live playback tracking
- *eve CLI* for transcription and rendering (see below)
- *hydra* (optional) for the =?= keybinding cheatsheet popup

The =eve= CLI ships as a Python package inside this repository.  On a fresh
clone it bootstraps itself automatically via =scripts/run-cli.sh= using the
pre-built wheel in =cli/dist/= — no =uv= or Python package manager required

#+begin_src shell

uv pip install cli/dist/eve_cli-*.whl
#+end_src

#+begin_src shell
eve doctor --json
#+end_src

(use-package eve
  :straight (:type git :host github :repo "behaghel/eve.el"))

Clone the repository, add it to =load-path=, and require it:

** Quick Start

This walkthrough takes you from a raw recording to a rendered clip in five
steps.

1. *Transcribe a recording*

   #+begin_example
   M-x eve-transcribe RET
   #+end_example

   Select the directory containing your =.mp4= files.  Eve starts
   =eve transcribe= in the background.  When it finishes, the manifest opens
   automatically in eve-mode.

2. *Read through the transcript*

   Navigate with =j= / =k=.  The buffer shows each spoken segment as a block
   of text.  Timestamps appear in the right margin.

3. *Delete unwanted content*

   Place point on a word and press =d= to delete it.  Press =D= to delete the
   whole segment.  Press =f= on a word to tag it as a filler; press =F= to
   delete all tagged fillers at once.

4. *Compile a draft preview*

   #+begin_example
   C-c C-c
   #+end_example

   Eve validates the manifest, pre-renders changed segments in the background,
   and produces a draft =.mp4= in seconds.  Play it back with
   =C-u C-c SPC= or your usual player.

5. *Export for delivery*

   #+begin_example
   C-u C-c C-c
   #+end_example

   Compiles at full H.264 quality using the same segment cache.  Only
   segments that haven't been cached yet need re-encoding.

* The TJM Manifest
:PROPERTIES:
:DESCRIPTION: Structure and representation of .tjm.json files.
:END:

** Structure

A TJM manifest is a JSON file with two top-level arrays:

- =sources= — a list of media files referenced by the edit.  Each entry has
  an =id= used throughout the manifest and a =file= path.

- =segments= — the ordered timeline.  Each segment references a source, a
  start and end time (in seconds or ticks), and the spoken text.  Optional
  fields carry per-word timestamps, b-roll metadata, speaker identity, tags,
  and notes.

A minimal manifest:

#+begin_src json
{
  "version": 1,
  "sources": [
    { "id": "clip01", "file": "raw/recording.mp4" }
  ],
  "segments": [
    {
      "id": "clip01-s0001",
      "source": "clip01",
      "start": 12.34,
      "end": 15.60,
      "text": "This is what was said."
    }
  ]
}
#+end_src

The TJM v1.1 specification in =docs/tjm-spec.md= defines every field,
including the tick-based timing model, the edit overlay, and the render
contract.

** How eve-mode renders a manifest

Eve-mode presents the manifest as readable text.  Each segment is shown as a
paragraph with its text content.  Markers appear as section headings.
Deleted words are hidden by default (toggle with =C-c C-h=).  Per-word
timestamps are hidden unless =C-c C-w= is active.

The buffer is read-only as prose — you interact through keybindings, not
direct text editing.  The underlying JSON is always kept in sync.

** The raw JSON companion buffer

Press =C-c C-o= (=eve-open-raw-json=) to open the underlying JSON in a
companion buffer.  This is useful for low-level inspection or edits that
the structured commands don't cover.  Saving the companion buffer reloads
the eve-mode view.

* Transcription
:PROPERTIES:
:DESCRIPTION: Generating manifests from audio and video files.
:END:

** Starting transcription from Emacs

Two commands launch transcription:

- =M-x eve-transcribe= prompts for a directory, scans for media files, and
  starts =eve transcribe= asynchronously.  On success the manifest opens
  automatically in eve-mode.

- =M-x eve-dired-transcribe= runs from Dired using the marked files (or the
  file at point when nothing is marked).

On failure, the =*eve transcribe*= buffer appears so you can inspect the error
output.

** Backends

=eve transcribe= supports three backends:

| Backend | Flag | Notes |
|---------+------+-------|
| =faster-whisper= | =--backend faster-whisper= | Default.  Fast, local, good accuracy. |
| =transformers= | =--backend transformers= | HuggingFace models.  Needs optional deps. |
| =nemo= | =--backend nemo= | NVIDIA NeMo.  Needs optional deps. |

The =--verbatim= flag (on by default in Emacs) biases =faster-whisper= toward
preserving disfluencies and filler words.  For =transformers= and =nemo=,
choose a verbatim-friendly model instead.

Good starting points:
- =transformers=: =nyrahealth/CrisperWhisper=
- =nemo=: =nvidia/parakeet-ctc-1.1b=

** Configuration

| Variable | Default | Description |
|----------+---------+-------------|
| =eve-transcribe-backend= | ="faster-whisper"= | Backend name passed to =eve transcribe=. |
| =eve-transcribe-model= | ="medium.en"= | Model identifier. |
| =eve-transcribe-verbatim= | =t= | Pass =--verbatim= to the backend. |
| =eve-transcribe-tag-fillers= | =t= | Tag filler words during transcription. |
| =eve-media-extensions= | =("mp4" "mov" ...)= | File types scanned by =eve-transcribe=. |

** Installing optional backends

uv sync --project cli --extra nemo
#+end_src

Or from =cli/=:

#+end_src

* Editing
:PROPERTIES:
:DESCRIPTION: Navigation, deletion, splitting, merging, and metadata.
:END:

** Navigation

| Key | Command | Description |
|-----+---------+-------------|
| =j= / =n= | =eve-next-segment= | Move to next segment. |
| =k= / =p= | =eve-previous-segment= | Move to previous segment. |

** Deletion and restoration

- =d= (=eve-delete-word=) — marks the word at point as deleted.  Deleted
  words are hidden in the buffer (=eve-hide-deleted-mode=) and excluded from
  compilation.  The operation is non-destructive: the word remains in the JSON
  and can be restored.

- =D= (=eve-delete-segment=) — marks the entire current segment as deleted.

Deleted content is shown or hidden with =C-c C-h= (=eve-hide-deleted-mode=).

** Splitting and merging

- =s= or =|= (=eve-split-segment=) — splits the current segment at point,
  creating two segments with the text and timing divided at the cut.

- =m= or =)= (=eve-merge-with-next=) — merges the current segment with the
  one below, combining text and adjusting the end time.

** Reordering

- =J= (=eve-move-segment-down=) — move the current segment one position down.
- =K= (=eve-move-segment-up=) — move the current segment one position up.

** Markers and sections

Press =o= (=eve-insert-marker=) to insert a marker above the current segment.
Markers act as named section headings.  They appear in the buffer as bold
titles.

When you compile with =C-c C-c= while point is on a marker, only the segments
in that section are compiled.  This is useful for working on one chapter of a
longer video.

** Inline editing

Press =e= (=eve-dwim-edit=) to edit the text or metadata of the segment at
point.  The command adapts to context: on a word it edits the spoken text; on
a marker it edits the title.

** Metadata

| Key | Command | Description |
|-----+---------+-------------|
| =t= | =eve-toggle-tag= | Toggle a tag on the segment at point. |
| =i= | =eve-edit-speaker= | Set the speaker label for the segment. |
| =r= | =eve-edit-start-end= | Edit the segment's start/end timecodes. |
| =C-c C-n= | =eve-edit-notes= | Edit free-form notes for the segment. |

** Undo and redo

- =u= (=eve-undo=) — undo the last edit.
- =C-r= (=eve-redo=) — redo a previously undone edit.

Eve maintains its own undo stack separate from Emacs's built-in undo.

** Validation

Eve validates the manifest automatically on every save (controlled by
=eve-validation-on-save=).  Manual validation runs with =C-c C-v=
(=eve-validate=).  Timing inconsistencies are reported in the echo area.

* Filler Words
:PROPERTIES:
:DESCRIPTION: Non-destructive tagging and deletion of filler words.
:END:

** Concept

Filler words ("um", "uh", "you know") clutter transcripts.  Eve's filler
workflow is non-destructive: words are /tagged/ as fillers first, then
/optionally deleted/.  Tagged fillers are highlighted in the buffer with a
distinct face and excluded from compilation.  You can untag them at any time.

** Configuration

#+begin_src emacs-lisp
;; Plain strings — single words or multi-word phrases
(setq eve-filler-phrases '("um" "uh" "you know" "to be honest with you"))
#+end_src

=eve-filler-phrases= stores phrases as plain strings.  Matching is
case-insensitive and ignores punctuation.  The list is persisted when you use
the interactive add commands.

The legacy =eve-filler-regex= variable is still honoured for backwards
compatibility but =eve-filler-phrases= is preferred.

** Commands

| Key | Command | Description |
|-----+---------+-------------|
| =f= | =eve-dwim-filler= | Add the word at point to phrases and retag; or delete region as filler. |
| =F= | =eve-delete-fillers= | Mark all tagged fillers as deleted. |
| =C-c C-d= | =eve-delete-fillers= | Same as =F= from the =C-c= prefix map. |
| — | =M-x eve-tag-fillers= | Tag all matching words in the buffer. |
| — | =M-x eve-add-filler-at-point= | Add the word at point to =eve-filler-phrases= and retag. |
| — | =M-x eve-add-filler-region= | Add the selected phrase to =eve-filler-phrases= and retag. |

** Auto-tagging on file open

Fillers are tagged automatically every time a TJM manifest is opened.  If you
add a new phrase to =eve-filler-phrases=, reopen the file or run
=M-x eve-tag-fillers= to apply the updated list.

* B-Roll
:PROPERTIES:
:DESCRIPTION: Attaching and managing supplementary video footage.
:END:

** Concept

B-roll is supplementary footage that plays over (or instead of) the main
recording.  Eve supports three compositing modes:

- *Replace* (default) — the b-roll clip completely replaces the source video
  for the duration of the segment.  The source audio is kept unless you change
  the audio policy.

- *Pip* (picture-in-picture) — the b-roll is scaled down and composited on
  top of the source video.  You control position and size.

- *Overlay* — alias for replace.  Treated identically by the renderer.

** Editing b-roll metadata

Press =b= (=eve-edit-broll=) on a segment to set or modify its b-roll.  The
command prompts for each field in turn:

| Field | Prompt | Description |
|-------+--------+-------------|
| =file= | "B-roll file" | Path to a video, image, or =.json= template.  Blank clears the b-roll. |
| =mode= | "Mode (overlay/replace/pip)" | Compositing mode (see above). |
| =audio= | "Audio (source/mute/broll)" | Which audio track the rendered segment uses. |
| =start_offset= | "Start offset (MM:SS or seconds)" | Where to begin playback in the b-roll clip. |
| =duration= | "Duration (MM:SS or seconds)" | How long the b-roll plays; blank uses the segment duration. |
| =still= | "Treat as still image?" | Auto-detected for =.png=, =.jpg=, etc.  Loops the image for the segment. |
| =continue= | "Continue from previous?" | Part of a continuation chain (see below). |

Additional keys:

| Key | Command | Description |
|-----+---------+-------------|
| =B= | =eve-toggle-broll-continue= | Copy b-roll from the previous segment and mark as continuation. |
| =C-c C-b= | =eve-edit-broll-placeholders= | Edit placeholder values for the current segment. |

B-roll metadata is stored in the segment's =edit.broll= field in the JSON
manifest.

** Compositing modes in detail

*** Replace mode

The b-roll video replaces the source video entirely.  The renderer trims the
b-roll to the segment's duration, starting at =start_offset=.  If the b-roll
is shorter than the segment, it is extended to fit.

When =still= is true, the image is looped for the full segment duration using
=ffmpeg -loop 1=.

*** Pip mode

The b-roll is scaled and positioned as an inset window over the source.
Position and size are controlled by a =position= object in the JSON:

#+begin_src json
{
  "file": "demo-recording.mp4",
  "mode": "pip",
  "position": {
    "x": 0.05,
    "y": 0.05,
    "width": 0.3
  }
}
#+end_src

| Field | Default | Description |
|-------+---------+-------------|
| =x= | =0.05= | Horizontal offset as a fraction of the main video width (0.0--1.0). |
| =y= | =0.05= | Vertical offset as a fraction of the main video height (0.0--1.0). |
| =width= | =0.3= | Width of the pip window as a fraction of the main video width. |

The height is calculated automatically to preserve the b-roll's aspect ratio.

** Audio policy

Each b-roll entry has an =audio= field that controls which audio track
appears in the rendered segment:

- =source= (default) — keep the original recording's audio.
- =broll= — use the audio track from the b-roll clip.  Not available for
  still images.
- =mute= — no audio for this segment.

** JSON templates

When the =file= field points to a =.json= file instead of a media file, eve
treats it as a *template*.  The template JSON has this structure:

#+begin_src json
{
  "template": "lower-third.png",
  "overlays": [
    {
      "placeholder": "SPEAKER_NAME",
      "fontsize": 36,
      "color": "white",
      "x": 80,
      "y": 620,
      "boxColor": "black@0.6"
    },
    {
      "placeholder": "SPEAKER_TITLE",
      "fontsize": 24,
      "color": "#cccccc",
      "x": 80,
      "y": 665
    }
  ],
  "placeholders": {
    "SPEAKER_NAME": "Full Name",
    "SPEAKER_TITLE": "Job Title"
  }
}
#+end_src

- =template= — path to the actual media file (image or video).
- =overlays= — a list of text overlays rendered with ffmpeg =drawtext=.
  Each overlay references a =placeholder= key by name.
- =placeholders= — default values for each placeholder.  Segment-level
  overrides (set via =C-c C-b=) take precedence.

*** Overlay fields

Each overlay in the =overlays= array supports:

| Field | Description |
|-------+-------------|
| =placeholder= | Name of the placeholder key this overlay renders. |
| =fontsize= (or =size=) | Font size in pixels. |
| =color= (or =fontColor=) | Font colour (ffmpeg colour syntax: name, =#rrggbb=, =white@0.8=). |
| =x= (or =position_x=) | Horizontal position (pixels or ffmpeg expression like =(w-text_w)/2=). |
| =y= (or =position_y=) | Vertical position (pixels or ffmpeg expression). |
| =align= | Set to ="center"= to auto-centre horizontally and vertically. |
| =font= (or =fontfile=) | Path to a =.ttf= or =.otf= font file. |
| =boxColor= (or =box_color=) | Background box colour (e.g. =black@0.6= for 60% opacity). |
| =shadowColor= / =shadow_x= / =shadow_y= | Drop shadow parameters. |
| =start= / =end= / =duration= | Time window (seconds) during which the overlay is visible. |

*** Concrete example: lower-third title card

Suppose you have a background image =lower-third.png= and you want to show
the speaker's name and title during a segment.

1. Create the template file =assets/lower-third.json=:

   #+begin_src json
   {
     "template": "assets/lower-third.png",
     "overlays": [
       {
         "placeholder": "NAME",
         "fontsize": 42,
         "color": "white",
         "x": 80,
         "y": 620,
         "font": "assets/Inter-Bold.ttf",
         "boxColor": "black@0.5"
       },
       {
         "placeholder": "ROLE",
         "fontsize": 28,
         "color": "#cccccc",
         "x": 80,
         "y": 672,
         "font": "assets/Inter-Regular.ttf"
       }
     ],
     "placeholders": {
       "NAME": "Speaker Name",
       "ROLE": "Speaker Role"
     }
   }
   #+end_src

2. In eve-mode, press =b= on the segment where you want the title card.
   Set =file= to =assets/lower-third.json=.  Set =mode= to =replace=.

3. Press =C-c C-b= to edit placeholders for this segment:

   #+begin_example
   Placeholder NAME (default Speaker Name): Jane Doe
   Placeholder ROLE (default Speaker Role): Product Lead
   New placeholder key (blank to finish):
   #+end_example

4. Compile with =C-c C-c=.  The rendered segment shows =lower-third.png=
   with "Jane Doe" and "Product Lead" burned in as text overlays at the
   specified positions and sizes.

Each segment can override different placeholder values from the same
template.  The template is reused; only the text changes.

** Placeholders without templates

You can also set placeholder key-value pairs directly on a segment without
a JSON template, using =C-c C-b=.  These are stored in the manifest under
=edit.broll.placeholders= and are available for any overlay definitions.

** Continuation chains

When a single b-roll clip should span multiple consecutive segments, use
continuation chains:

1. Press =b= on the first segment and configure the b-roll normally.
2. Move to the next segment and press =B= (=eve-toggle-broll-continue=).

=B= copies the b-roll metadata from the nearest preceding segment that has
b-roll, then auto-calculates:

- =continue=: set to =true=.
- =start_offset=: previous offset + previous segment duration, so playback
  continues seamlessly from where the last segment left off.
- =duration=: remaining duration of the b-roll clip (if the original had a
  duration set).

Repeat =B= on each subsequent segment in the chain.  The renderer treats the
entire chain as one continuous b-roll playback — the clip advances smoothly
across segment boundaries without restarting.

* Compilation and Rendering
:PROPERTIES:
:DESCRIPTION: Turning the manifest into a video file.
:END:

** Two modes: draft and final

=C-c C-c= (=eve-compile=) compiles the manifest at draft quality — fast
enough to catch mistakes before committing to a full export.

=C-u C-c C-c= compiles at final quality (H.264 medium/CRF 18) for delivery.

The quality used can be changed permanently with =eve-compile-quality=.

** Under the hood

Eve's rendering pipeline is designed to be fast after the first run:

- *Quality profiles* — draft uses MJPEG encoding at ultrafast preset; final
  uses H.264 at medium preset.  The codec and preset are resolved
  automatically from the =--quality= flag.

- *Parallel encoding* — segments are encoded concurrently across all available
  CPU cores (=eve-compile-jobs=).

- *Segment cache* — rendered segments are stored in =.eve-cache/= next to the
  manifest, keyed by a SHA-256 hash of the segment's content and encoding
  settings.  Unchanged segments are reused; only changed ones re-encode.

- *Speculative pre-rendering* — after each save, an idle timer
  (=eve-auto-render-idle-seconds=) fires a background render of the segments
  that changed.  When you press =C-c C-c=, the cache is already warm and
  compilation reduces to a concat step.

** Pre-flight validation

Before starting a render, eve validates the manifest:

- All source files must exist and be readable.
- All b-roll files must exist.
- Segment timecodes must be non-negative and end after start.
- Words must fall within their segment bounds.

Warnings appear in the echo area.  Errors block compilation unless you
confirm.  Run validation explicitly with =C-c C-v= (=eve-validate=).

If the estimated render time (from the dry-run analysis) exceeds
=eve-compile-confirm-threshold= seconds, eve asks for confirmation.

** Dry-run analysis

Before rendering, eve can report which segments would be re-encoded and
estimate the render time:

#+begin_src shell
eve text-edit manifest.tjm.json --output out.mp4 --dry-run
#+end_src

The output (JSON with =--json=) includes =total_segments=,
=changed_segments=, and =estimated_seconds=.  This information feeds the
pre-flight confirmation prompt.

** Checkpoint and resume

If a render is interrupted, eve writes a checkpoint file in =.eve-cache/= as
segments complete.  The next run picks up from where it left off: segments
already in the cache are reused, and only the remaining ones are encoded.

Pass =--no-resume= to force a fresh render from scratch.

** Progressive output

With =--partial-every N=, eve writes a partial =<output>.partial.mp4= after
every N segments complete.  A JSON event is emitted on stdout for each
partial file, which Emacs can use to update a live preview.

** Configuration reference

| Variable | Default | Description |
|----------+---------+-------------|
| =eve-compile-quality= | ="draft"= | ="draft"= for fast preview, ="final"= for delivery. |
| =eve-compile-jobs= | =0= | Parallel encode workers; 0 = auto-detect CPU count. |
| =eve-compile-scale= | =1.0= | Resolution scale for draft (e.g. =0.5= = half res). |
| =eve-compile-cache= | =t= | Use the =.eve-cache/= segment cache. |
| =eve-compile-confirm-threshold= | =30= | Estimated seconds above which a confirmation prompt appears. |
| =eve-auto-render= | =t= | Speculatively pre-render changed segments after each save. |
| =eve-auto-render-idle-seconds= | =2.0= | Idle delay before the speculative render fires. |
| =eve-play-after-compile= | =nil= | When non-nil, play the output automatically after compilation. |
| =eve-preserve-gaps-max= | =2.0= | Maximum gap (seconds) that is preserved between segments. |

** CLI reference

Under normal Emacs usage you never need to call =eve text-edit= directly —
the Emacs commands handle all flags.  For scripting or CI:

#+begin_example
eve text-edit manifest.tjm.json --output out.mp4 [FLAGS]

--quality draft|final   Encoding profile (default: draft)
--codec h264|mjpeg      Override intermediate codec
--scale FACTOR          Resolution scale (e.g. 0.5)
--jobs N                Parallel workers (0 = auto)
--cache-dir PATH        Segment cache location (default: .eve-cache/)
--no-cache              Disable segment cache
--cache-max-size GB     Cache eviction threshold (default: 10)
--validate              Check manifest without rendering
--dry-run               Report cache hits/misses and estimated time
--segments ID [ID …]    Pre-render only listed segments into cache
--no-resume             Ignore checkpoint from a prior interrupted render
--partial-every N       Write a partial .mp4 every N segments
#+end_example

* Playback
:PROPERTIES:
:DESCRIPTION: Live preview of source and rendered video.
:END:

** Source playback

=C-c SPC= (=eve-play=) or =SPC= / =RET= (=eve-play-segment=) plays the
source media from the current segment to the end of the clip.  All segments
from the same source file are tracked, so the playback highlight follows even
across segments.

Playback position maps directly to the original recording timestamps.

** Rendered playback

=C-u C-c SPC= (=eve-play= with prefix) or =C-c C-p= (=eve-play-rendered=)
plays the compiled output file.  If the output is missing or older than the
manifest, eve compiles it first.

Progress is tracked using the post-edit rendered timeline; deleted content is
skipped in the position calculation.

** Playback tracking and segment highlighting

Eve integrates with mpv via its JSON IPC socket.  The currently-playing
segment is highlighted with a vivid amber overlay (=eve-playback-face=),
distinct from both the focus highlight and the region.

** Controls

| Key | Action |
|-----+--------|
| =SPC= (during playback) | Pause / resume. |
| Move point to a segment | Seek mpv to that segment's start time. |
| =C-c k= / =M-x eve-stop-playback= | Stop playback and restore keybindings. |
| =<left>= | Seek backward =eve-seek-short-seconds= (default 5s). |
| =<right>= | Seek forward =eve-seek-short-seconds= (default 5s). |
| =S-<left>= | Seek backward =eve-seek-long-backward-seconds= (default 30s). |
| =S-<right>= | Seek forward =eve-seek-long-forward-seconds= (default 10s). |

Stopping playback restores all keybindings to their normal state.

** Video layout

Set =eve-video-layout= to =t= to arrange the screen for editing: Emacs takes
the bottom portion of the display and mpv sits directly above it —
borderless, always-on-top, with automatic letterboxing.

#+begin_src emacs-lisp
(setq eve-video-layout t)

;; Fraction of screen height for the video window (default 0.6 = 60%)
(setq eve-video-layout-ratio 0.6)
#+end_src

The layout activates on the first play command and the original frame geometry
is restored when playback stops.  Skipped when Emacs is in macOS native
fullscreen.

Configuration:

| Variable | Default | Description |
|----------+---------+-------------|
| =eve-play-program= | ="mpv"= | Player binary. |
| =eve-play-args= | =(list "--quiet" ...)= | Extra arguments passed to the player. |
| =eve-video-layout= | =nil= | Enable automatic screen layout on play. |
| =eve-video-layout-ratio= | =0.6= | Fraction of screen height for the video. |
| =eve-seek-short-seconds= | =5= | Short seek step in seconds. |
| =eve-seek-long-backward-seconds= | =30= | Long backward seek step in seconds. |
| =eve-seek-long-forward-seconds= | =10= | Long forward seek step in seconds. |

* Display
:PROPERTIES:
:DESCRIPTION: Timestamp ruler, hidden segments, and word-level view.
:END:

** Timestamp ruler

=eve-ruler-mode= (active by default) displays =[hh:mm:ss]= markers in the
right margin.  Each marker is placed next to the segment that /reaches/ that
milestone in the rendered video — deleted content is excluded from the running
total.

The total rendered duration appears in the mode line.

#+begin_src emacs-lisp
;; Change the milestone interval (default 30 seconds)
(setq eve-ruler-interval 60.0)
#+end_src

Toggle the ruler with =C-c C-l= (=eve-ruler-mode=).

** Hiding deleted segments

=C-c C-h= (=eve-hide-deleted-mode=) toggles visibility of deleted words and
segments.  When active (the default), deleted content is hidden and the
transcript reads as it will render.  When inactive, deleted content appears
with a strikethrough face.

** Word-level display

=C-c C-w= (=eve-toggle-words=) shows or hides per-word timing annotations
under each segment.  Useful when inspecting or adjusting precise cut points.

| Variable | Default | Description |
|----------+---------+-------------|
| =eve-ruler-interval= | =30.0= | Seconds between ruler markers. |
| =eve-show-words-by-default= | =nil= | Show word timings on buffer open. |
| =eve-validation-on-save= | =t= | Validate manifest on every save. |
| =eve-validation-time-tolerance= | =0.1= | Timing slack tolerated during validation (seconds). |

* Architecture
:PROPERTIES:
:DESCRIPTION: Two-layer design and the Emacs–CLI contract.
:END:

** Two-layer boundary

Eve is deliberately split into two independent layers:

- *The Emacs package* (=eve.el=) owns the user interface: major mode,
  interactive editing commands, playback integration, asynchronous process
  launching, and parsing of CLI output.  It does not implement media-processing
  logic.

- *The CLI* (=cli/=, Python) owns all media processing: manifest generation,
  denoise orchestration, filler policy, timeline construction, and FFmpeg
  invocation.  It does not embed Emacs-specific behaviour.

This boundary means the CLI can be used standalone (e.g. in CI pipelines) and
the Emacs package can evolve independently.

** Shared contract

The two layers communicate through:

- *TJM files on disk* — the =.tjm.json= format defined in =docs/tjm-spec.md=.
- *Machine-readable JSON output* — the CLI emits structured JSON when called
  with =--json=; the Emacs package parses it.
- *Stable exit codes* — 0 for success, non-zero for failure.

** devenv commands

The project uses [[https://devenv.sh][devenv]] at the repository root.

#+begin_src shell
# Emacs
devenv shell -- format-elisp   # Format .el files
devenv shell -- parse           # Check Emacs Lisp syntax
devenv shell -- checkdoc        # Lint docstrings
devenv shell -- load-check      # Verify the package loads
devenv shell -- compile         # Byte-compile
devenv shell -- test-elisp      # Run ERT tests

# Python CLI
devenv shell -- sync-py         # Install Python deps
devenv shell -- format-py       # Format Python code
devenv shell -- lint-py         # Run ruff / mypy
devenv shell -- test-py         # Run pytest
devenv shell -- build-py        # Build the wheel
devenv shell -- run-cli -- --help

# Combined
devenv shell -- format          # Format all
devenv shell -- lint            # Lint all
devenv shell -- test            # Test all
devenv shell -- ci              # Full CI pipeline

# Documentation
devenv shell -- docs            # Generate eve.texi and eve.info
#+end_src

* Appendix A: TJM v1.1 Specification
:PROPERTIES:
:DESCRIPTION: Reference to the normative TJM format specification.
:APPENDIX: t
:END:

The normative TJM v1.1 specification is maintained in =docs/tjm-spec.md= in
the repository.  It defines:

- The complete JSON schema for =.tjm.json= files.
- The tick-based timing model and its relationship to real-time seconds.
- The edit overlay structure (=edit= field on segments and words).
- The deterministic render contract that =eve text-edit= implements.
- Migration paths from earlier TJM versions.

To read the specification, open =docs/tjm-spec.md= in the repository or visit
the repository at [[https://github.com/behaghel/eve.el]].
